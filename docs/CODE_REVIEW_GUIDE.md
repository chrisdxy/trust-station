# 代码审查标准与流程指南

## 📋 代码审查标准 (Code Review Standards)

### 🎯 审查目标

- **提高代码质量**：确保代码正确、可读、可维护
- **知识共享**：团队成员互相学习最佳实践
- **发现缺陷**：在合并前发现潜在问题
- **保持一致性**：确保代码风格和架构决策一致

---

## ✅ 代码审查清单 (Review Checklist)

### 🔴 必须修复 (Blockers)

审查者必须检查以下各项，**任一项不通过不能合并**：

#### 1. 功能正确性

- [ ] 代码实现了需求/修复了 bug
- [ ] 边界条件处理正确（空值、异常输入、并发）
- [ ] 错误处理完善（try-catch、错误提示、日志记录）
- [ ] 没有硬编码的敏感信息（密码、API key、token）

#### 2. 安全性

- [ ] 输入验证：所有用户输入都经过验证和清理
- [ ] SQL 注入防护：使用参数化查询（prepared statements）
- [ ] XSS 防护：React 自动转义 JSX，但 dangerouslySetInnerHTML 需审查
- [ ] 认证/授权：敏感操作检查用户权限
- [ ] API 端点：验证 token、检查角色权限

#### 3. 性能

- [ ] 没有明显的性能问题（N+1 查询、无限循环、内存泄漏）
- [ ] 数据库查询有索引支持
- [ ] 大列表使用分页或虚拟滚动
- [ ] 图片/资源有懒加载或优化

#### 4. 可维护性

- [ ] 函数长度 ≤ 50 行（超过需要拆分）
- [ ] 文件长度 ≤ 300 行（超过需要拆分）
- [ ] 复杂逻辑有注释说明
- [ ] 没有重复代码（遵循 DRY 原则）

---

### 🟡 应该修复 (Suggestions)

审查者**建议**修复，但作者可以讨论：

#### 1. 代码风格

- [ ] 变量/函数命名清晰、符合语义
- [ ] 没有魔法数字（使用常量）
- [ ] 代码结构清晰（早期返回、减少嵌套）
- [ ] TypeScript 类型定义完整（不使用 `any`）

#### 2. 架构设计

- [ ] 组件职责单一（单一职责原则）
- [ ] API 路由遵循 RESTful 规范
- [ ] 共享逻辑提取到 hooks/utils/services
- [ ] 没有过度抽象（YAGNI 原则）

#### 3. 测试

- [ ] 关键功能有单元测试
- [ ] 新 API 有集成测试
- [ ] 边界情况有测试用例

---

### 💭 可选改进 (Nits)

审查者提出，作者自行决定是否采纳：

- [ ] 代码风格细节（空格、换行、引号）
- [ ] 变量/函数重命名建议
- [ ] 注释润色
- [ ] 性能微优化

## 🔧 自动化检查 (Automated Checks)

以下检查应通过 **CI/CD** 自动执行：

### 1. Linting & Formatting

```bash
# ESLint 检查
npm run lint

# TypeScript 类型检查
npm run ts-check

# Prettier 格式化（可选）
npx prettier --check "src/**/*.{ts,tsx}"
```

### 2. 构建检查

```bash
# 确保项目可以构建
npm run build
```

### 3. 单元测试（如果已配置）

```bash
npm test
```

**规则**：所有自动化检查必须通过，否则 PR 不能合并。

## 📝 代码审查流程 (Review Process)

### 1. 开发者提交 PR 前

- [ ] 自审代码（把自己当作审查者）
- [ ] 运行所有自动化检查（lint、type-check、build）
- [ ] 填写 PR 描述模板（见下）
- [ ] 添加截图/录屏（如果是 UI 变更）

### 2. 分配审查者

- **必须**：至少 1 名核心成员审查
- **建议**：涉及特定模块时，添加该模块负责人
- **可选**：大型 PR（>500 行）需要 2 名审查者

### 3. 审查者响应时间

- **目标**：24 小时内首次响应
- **最长**：48 小时内必须响应
- **紧急 PR**：标注 `[URGENT]`，4 小时内响应

### 4. 审查反馈规范

审查者应使用以下标记：

```
🔴 BLOCKER: 必须修复才能合并
🟡 SUGGESTION: 建议修复，可讨论
💭 NIT: 小改进，可选

示例：
🔴 BLOCKER: 这里缺少输入验证，可能导致 SQL 注入
🟡 SUGGESTION: 这个函数可以拆分成 3 个更小的函数
💭 NIT: 变量名 `d` 可以改成 `userData` 更清晰
```

### 5. 作者修订代码

- 回复每条反馈（说明如何修复或为什么不修复）
- 推送新提交后，在 PR 中添加评论 "@reviewer 已修复"
- 不要 force push（保持审查历史）

### 6. 审查通过 & 合并

- 所有 🔴 BLOCKER 都已修复
- 所有自动化检查通过
- 至少 1 名审查者批准（Approved）
- 合并方式：
  - **小型 PR**（<100 行）：Squash and merge
  - **大型 PR**（>100 行）：Rebase and merge（保持提交历史）

## 🚨 常见审查问题 & 解决方案

### 问题 1：审查意见过多（>10 条）

**解决**：

- 审查者应先指出 🔴 BLOCKER，其他建议批量提出
- 作者先修复 BLOCKER，再处理建议
- 拆分 PR（一个 PR 只做一件事）

### 问题 2：审查者长期不响应

**解决**：

- 友好提醒（"Hi @reviewer，方便审查一下吗？"）
- 超过 48 小时，可请求其他审查者
- 紧急 PR 直接 @ 团队负责人

### 问题 3：作者和审查者意见不合

**解决**：

- 在 PR 评论中讨论（不要私下解决）
- 引用文档/最佳实践支持自己的观点
- 无法达成一致时，请求第三方仲裁（团队负责人）

## 📊 代码审查指标 (Metrics)

定期（每月）回顾以下指标，持续改进：

| 指标                          | 目标      | 说明       |
| ----------------------------- | --------- | ---------- |
| PR 从提交到首次审查的平均时间 | < 24 小时 | 响应速度   |
| PR 从提交到合并的平均时间     | < 3 天    | 交付速度   |
| 每个 PR 的平均审查者数量      | ≥ 1       | 知识共享   |
| 每个 PR 的平均评论数          | 3-7 条    | 审查深度   |
| 审查后发现的 bug 比例         | < 5%      | 审查有效性 |
